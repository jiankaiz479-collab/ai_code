# Remove BG Pipeline — 升級路線圖

> 紀錄「衣服去背」這個功能遇到的問題，以及每一版實際解決了什麼。

---

## 背景

最初版本（舊版）的去背流程很單純：把使用者上傳的圖丟給 `rembg` 套件（用 `u2net_human_seg` 模型），得到的結果直接回給前端。

這個流程在「網路上抓的乾淨衣服圖」可以動，但**遇到現場用手機拍的照片就會壞掉**。我們從這裡開始升級。

---

## v1 — 讓「手機現場拍的照片」能用 ✅（已上線）

### v1 遇到的問題

1. **使用者拿 iPhone 拍照後上傳，伺服器讀不到圖**
   - iPhone 預設存成 `.HEIC` 格式，Pillow 預設不支援。

2. **就算讀到圖，照片方向也是錯的**
   - 手機相簿打開照片是直的，但傳到伺服器後變成「躺著」的。
   - 原因：手機拍照時其實是用橫向方式記錄像素，靠檔案裡一個小標籤註明「請看的程式幫我旋轉 90 度」。相簿會自動處理，但 Python 預設不會。
   - 結果：丟給 AI 的是橫躺的衣服，模型判斷會出錯。

3. **iPhone 拍的顏色，伺服器看到的會偏掉**
   - iPhone 拍照用的是「比較鮮豔的顏色規格」(P3)，但伺服器與 AI 預設用「標準顏色規格」(sRGB) 解讀。
   - 比喻：就像 iPhone 用美金記帳，伺服器卻直接把數字當台幣處理，數字一樣但意思差很多。
   - 結果：使用者拍的鮮豔橘 T-shirt，AI 看到的可能變成偏紅的橘 → 後續顏色判斷、3D 上色都會偏。

4. **拍 4032×3024 的大圖，rembg 跑很慢、邊緣還會抖**

5. **rembg 預設模型是 `u2net_human_seg`，這是給「人」用的**
   - 但這個 endpoint 是「衣服去背」，輸入是「攤平的衣服」不是「人」，模型本來就不對。

6. **去背後的圖邊緣很髒**
   - 邊緣鋸齒、白邊 fringe。
   - 鞋子、地板雜物等「不是主體」的東西也被當前景留下來。

7. **AI 偷懶交差，系統還以為它成功了**
   - 去背的本質 = AI 要把圖切成「保留的部分（衣服）」和「刪掉的部分（背景）」。正常情況下，保留的應該佔整張圖的 30~80%。
   - 但有兩種偷懶方式 AI 會做：
     - **(A) 整張圖都保留**（佔 99%）→ 等於沒去背，跟原圖一樣
     - **(B) 整張圖都刪掉**（佔 1%）→ 回給前端一張幾乎全透明的圖
   - AI 偷懶時**程式碼從沒拋例外**，所以系統以為「任務完成」，回傳 HTTP 200。
   - **實例**：對照組 #1 那張凌晨拍的暗光照（水泥地 + 金屬櫃 + 塑膠袋），AI 判斷不出主體，乾脆把整張圖都當成「要保留」（情境 A），回給前端一張看起來和原圖一模一樣的 PNG。使用者按了「去背」按鈕，結果什麼都沒發生 —— 但系統那邊紀錄是「成功」。

### v1 做了什麼

| 問題 | 解法 |
|---|---|
| 1 | 支援格式擴充到 **HEIC / HEIF / JPG / JPEG / PNG / WebP / BMP / TIFF**。 |
| 2 | 讀檔時自動套 EXIF 旋轉旗標。 |
| 3 | 偵測 ICC profile，轉成 sRGB。 |
| 4 | 長邊超過 2048px 自動縮小。 |
| 5 | rembg 模型換成 `isnet-general-use`。 |
| 6 | 加 alpha matting + 最大連通區域過濾 + 邊緣收 1px。 |
| 7 | AI 跑完後，先算「保留的部分佔整張圖幾 %」：>95% 或 <3% 就判定「AI 偷懶了」，主動回失敗、附上原因。**限制**：只能抓兩種極端偷懶；如果 AI 找到一塊「形狀看起來像物體、佔 20% 左右」的區域（例如把筆電螢幕邊框當衣服），這個檢查抓不到，要 v4 用 LLM 預檢才能擋。 |

### 順帶做的架構安排

舊版去背流程已經有人在用，不能直接砍掉。所以用 **Strategy Pattern** 把舊流程包成 `LegacyRemoveBg`、新流程包成 `RobustRemoveBg`，靠 `.env` 的 `REMOVE_BG_VERSION` 切換。

這樣可以：
- 任何時候 rollback 回舊版
- A/B 比對兩版差異
- 之後加 v3 / v4 都不用動現有 code

### v1 之後仍未解決的問題

- **「上傳的不是衣服」抓不到** → mask 面積檢查只能抓「完全沒前景」或「整張都前景」兩種極端。如果使用者拍筆電螢幕、寵物、食物、UI 截圖等，rembg 仍會找到某個「像物體」的連通區域，mask 面積落在合理範圍（例如 20%），系統當作成功放行，產出一張無意義的 PNG。要 v4 用 Gemini 預檢「這是不是衣服」才能擋。
- 衣服掛在衣架上拍 → 衣架會跟衣服一起被保留（因為兩者像素相連，「最大連通區域」過濾不掉）。
- 極端暗光 + 雜亂背景 → 模型過度保守，幾乎整張都判為前景。
- 失敗時拿不到原始輸入，工程師難以重現問題。
- 「mask 面積超過 95% 才算失敗」這個閾值可能太寬鬆。

## v2 — 把爛輸入擋下來、把失敗存起來 🔴（規劃中，1–2 週）

### v2 要解決的問題

1. **使用者上傳的根本不是衣服**（v1 最大破口）
   - 拍到食物、寵物、風景、截圖、UI 螢幕、筆電 —— v1 仍會嘗試去背，產出一張無意義的 PNG。
   - 純像素檢查（v1 的佔比檢查）抓不到，因為這種輸入「看起來合理」。

2. **輸入照片本身就壞，模型再強也救不回來**
   - 模糊、過曝、過暗的照片送進 rembg，輸出的 mask 一定不準。
   - 目前沒有任何輸入品質檢查。

3. **線上跑出失敗時，無法重現**
   - 使用者抱怨「我那張圖去背失敗」，但伺服器沒留下那張原圖，工程師無法 debug。

4. **失敗回傳訊息太籠統**
   - 目前失敗只回「1500 處理失敗」，前端與使用者都不知道為什麼。

### v2 預計怎麼做

| 問題 | 解法 |
|---|---|
| 1 | 進 rembg 之前先丟 Gemini 預檢：「這張圖主體是不是衣服？」回答不是就直接擋，回 `1421 主體非衣服` + suggest「請上傳衣服照片」。 |
| 2 | 加品質檢查：Laplacian 變異數測模糊度、直方圖測曝光。任一不過就直接擋。 |
| 3 | 失敗時把原圖存到 `tests/fail_samples/{日期}/{錯誤碼}_{hash}.jpg`。 |
| 4 | 把錯誤碼細分：1421（不是衣服）、1422（圖太糊/太暗）、1500（其他）等。 |

### 順手做掉的小事

- 把 `REMOVE_BG_MASK_COVERAGE_MAX` 從 `0.95` 收緊到 `0.85`。

### 為什麼提前到 v2

原本「Gemini 預檢」排在 v4，但實測發現 v1 的最大破口就是這個 —— 例如有人上傳筆電螢幕，系統還是會「成功」交出一張黑邊框的 PNG。這種錯誤對使用者最致命（按了按鈕，得到完全錯的結果），而且擋下來的成本只是「多一次 Gemini API 呼叫」，與其等 v4 不如先做。

## v3 — 處理「人穿著衣服」的拍法 🟡（中期，2–4 週）

### v3 要解決的問題

1. **目前完全沒辦法處理「人穿著拍照」的情境**
   - 現在的 robust 流程假設前景就是「衣服本體」。
   - 但實務上很多人會「穿著要試的衣服自拍」上傳，這時前景是「人 + 衣服」，要的卻只是「衣服那一塊」。

2. **形狀明顯不像衣服的東西，目前還是會通過**
   - 例如對照組 #4 那張：粉紅衣架跟白短褲一起被保留。
   - 「最大連通區域」只能擋「不相連」的雜物，擋不了「相連但形狀詭異」的東西。

### v3 預計怎麼做

| 問題 | 解法 |
|---|---|
| 1 | 加人體部位分割（SCHP, Self-Correction Human Parsing）。模型輸出「上衣 / 褲子 / 裙子」獨立 mask，使用者拍人穿著的照片也能單獨抽衣服。 |
| 2 | 在 mask 後處理階段加形狀檢查：bbox 長寬比、主軸方向。明顯細長條的（衣架）就標記異常並拒絕。 |

### 順帶會解掉的 edge case

- 同一張照片裡同時有上衣和褲子 → 部位分割天然能分開輸出。
- 黑衣黑桌 / 白衣白牆等極端對比情境 → 如果 SCHP 仍判不準，加 `birefnet-general` 當 fallback。

---

## v4 — 處理罕見但真實的 edge case 🟢（長期，1–2 個月）

### v4 要解決的問題

1. **單一模型在 edge case 偶爾失效**
   - 半透明衣服（紗裙、漁網衣）、強烈反光（皮衣絲質）、衣服折疊堆疊。
   - 這些靠單一模型很難穩定處理。

### v4 預計怎麼做

| 問題 | 解法 |
|---|---|
| 1 | 多模型 ensemble（u2net + isnet + birefnet 投票），或 SAM 2 + 提示點互動式分割。 |

---

## v5 — 維運基礎建設（與 v2~v4 並行進行）⚪

不是新功能，是讓系統能長期穩定運作的事：

- 每個階段加上耗時 log，知道哪一步慢。
- 圖太大或併發太高會吃光記憶體 → 加 Celery queue + 上傳大小上限。
- 沒有任何成功率 / 失敗率的統計 → 寫個 SQLite 表記錄每次請求結果。

| 版本 | 主題 | 解決的問題 | 狀態 |
|---|---|---|---|
| v1 | 讓手機拍的照片能用 | 7 個 | ✅ 已上線 |
| v2 | LLM 預檢 + 擋爛輸入 + 失敗可重現 | 4 個 | 🔴 規劃中 |

## 排序的理由

- **v2 先做**：包含 v1 最大破口（非衣服輸入）+ 廉價的品質檢查。多一次 Gemini 呼叫成本不高，但擋掉了「使用者按按鈕卻得到錯結果」的最致命錯誤。
- **v3 排在 v2 之後**：輸入治好之後，模型升級才能真正展現效果。輸入髒的時候換什麼模型都沒用。
- **v4 排最後**：半透明、反光、堆疊這類 case 罕見且難解，要先確認前面三版搞定常見場景才付這個錢。
- **v5 並行**：沒有觀測性指標，前面所有改動都不知道有沒有效。

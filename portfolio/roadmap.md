# Remove BG Pipeline — 升級路線圖

> 紀錄「衣服去背」這個功能遇到的問題，以及每一版實際解決了什麼。

---

## 背景

最初版本（舊版）的去背流程很單純：把使用者上傳的圖丟給 `rembg` 套件（用 `u2net_human_seg` 模型），得到的結果直接回給前端。

這個流程在「網路上抓的乾淨衣服圖」可以動，但**遇到現場用手機拍的照片就會壞掉**。我們從這裡開始升級。

---

## v1 — 讓「手機現場拍的照片」能用 ✅（已上線）

### v1 遇到的問題

1. **使用者拿 iPhone 拍照後上傳，伺服器讀不到圖**

   - iPhone 預設存成 `.HEIC` 格式，Pillow 預設不支援。

2. **就算讀到圖，照片方向也是錯的**

   - 手機相簿打開照片是直的，但傳到伺服器後變成「躺著」的。

   - 原因：手機拍照時其實是用橫向方式記錄像素，靠檔案裡一個小標籤註明「請看的程式幫我旋轉 90 度」。相簿會自動處理，但 Python 預設不會。

   - 結果：丟給 AI 的是橫躺的衣服，模型判斷會出錯。

3. **iPhone 拍的顏色，伺服器看到的會偏掉**

   - iPhone 拍照用的是「比較鮮豔的顏色規格」(P3)，但伺服器與 AI 預設用「標準顏色規格」(sRGB) 解讀。

   - 比喻：就像 iPhone 用美金記帳，伺服器卻直接把數字當台幣處理，數字一樣但意思差很多。

   - 結果：使用者拍的鮮豔橘 T-shirt，AI 看到的可能變成偏紅的橘，後續顏色判斷、3D 上色都會偏。

4. **拍 4032×3024 的大圖，rembg 跑很慢、邊緣還會抖**

5. **rembg 預設模型是 `u2net_human_seg`，這是給「人」用的**

   - 但這個 endpoint 是「衣服去背」，輸入是「攤平的衣服」不是「人」，模型本來就不對。

6. **去背後的圖邊緣很髒**

   - 邊緣鋸齒、白邊 fringe。

   - 鞋子、地板雜物等「不是主體」的東西也被當前景留下來。

7. **AI 偷懶交差，系統還以為它成功了**

   - 去背的本質是 AI 要把圖切成「保留的部分（衣服）」和「刪掉的部分（背景）」。正常情況下，保留的應該佔整張圖的 30%~80%。

   - 但有兩種偷懶方式 AI 會做：

     - **(A) 整張圖都保留**（佔 99%），等於沒去背，跟原圖一樣。

     - **(B) 整張圖都刪掉**（佔 1%），回給前端一張幾乎全透明的圖。

   - AI 偷懶時程式碼不一定拋例外，所以系統可能誤判「任務成功」。

### v1 做了什麼

| 問題 | 解法 |

|---|---|

| 1 | 支援格式擴充到 **HEIC / HEIF / JPG / JPEG / PNG / WebP / BMP / TIFF**。 |

| 2 | 讀檔時自動套 EXIF 旋轉旗標。 |

| 3 | 偵測 ICC profile，轉成 sRGB。 |

| 4 | 長邊超過 2048px 自動縮小。 |

| 5 | rembg 模型換成 `isnet-general-use`。 |

| 6 | 加 alpha matting + 最大連通區域過濾 + 邊緣收 1px。 |

| 7 | AI 跑完後先算保留比例：>95% 或 <3% 判定失敗並附原因。 |

### 順帶做的架構安排

舊版去背流程已經有人在用，不能直接砍掉。所以用 **Strategy Pattern** 把舊流程包成 `LegacyRemoveBg`、新流程包成 `RobustRemoveBg`，靠 `.env` 的 `REMOVE_BG_VERSION` 切換。

### v1 之後仍未解決的問題

- **「上傳的不是衣服」抓不到**：面積檢查只能抓到兩種極端情況。

- 衣服掛在衣架上拍時，衣架常跟衣服一起被保留。

- 極端暗光 + 雜亂背景時，模型可能過度保守。

- 失敗時拿不到原始輸入，工程師難以重現。

## v2 — LLM 預檢、擋爛輸入、可重現失敗 ✅（已上線）

### v2 目標問題（精簡）

1. 非衣服類輸入通過（筆電螢幕、海報、衣架、雜物等）。

2. 模型產生誤判但 API 仍回 200、缺少明確錯誤碼與使用者指引。

> 註：原本「失敗不可重現或缺少可重現樣本」先暫緩，不納入本期 v2 實作。

### v2 要做的重點

- **插入輕量 InputValidation 層（在去背策略之前）**：在 v1 的前處理基礎上，做快速視覺 heuristics（縮圖邊緣/模糊/曝光/長寬比）；可疑時才呼叫 LLM 做語意預檢。

- **多指標綜合決策**：結合 mask 面積、最大連通區域形狀、LLM 分數、分割置信度，而非只靠單一閾值。

- **錯誤碼與 API 行為**：新增 `14xx`（validation）與 `15xx`（LLM/semantic）分群，回 4xx 並附標準化導引。

- **指標與監控**：記錄 `validation_passed`、`validation_reason`、`final_status`。

> 註：失敗樣本留存與完整事後追查能力（原圖/mask/log/metadata）改到後續版本處理。

### 簡短執行步驟（短期）

1. 實作 `InputValidation` prototype（lightweight heuristics + LLM gate）。

2. 在去背流程最前面加一層 InputValidation gate，放在 `RemoveBgStrategy` 之前；走 `robust_v2` 時就先呼叫共享 `validators`。

3. 建立 `14xx/15xx` 錯誤碼對應與 API 回應導引。

4. 小流量（5%~10%）上線，收集誤放行率，微調閾值。

5. 達標後擴大流量並準備 v3。

## v3 — 人體部位分割與雙軌路由（人穿著情境）🟡（中期）

### v3 遇到的問題

1. **使用者上傳「真人穿搭照」時，rembg 會把整個人切下來**
   - `rembg` 是二元分割模型，對它來說「人體」就是前景。
   - 結果：無法單獨切出「衣服」，手腳與臉部都會殘留。

2. **系統無法自動判斷輸入情境，單一模型必定在某處失效**
   - 系統不知道使用者上傳的是「平鋪 / 衣架」還是「真人穿著」。
   - 如果全面換成服裝分割模型，遇到「平鋪在床上的衣服」會因為找不到人體而導致辨識失敗。

### v3 預計做什麼

| 問題 | 解法 |
|---|---|
| 1 | **引入 Human Parsing (SCHP)**：多類別語意分割模型。保留標籤為 5(上衣) 或 9(褲子) 的像素，人體與背景直接濾除，一次完成去背與提取。 |
| 2 | **LLM 路由 (Router) + 雙軌並行**：擴充 v2 的 Gemini 預檢，判斷 `presentation_mode`。平鋪圖走 `rembg`；真人圖走 `Human Parsing`。 |

### 順帶做的 UX 與架構安排

- **全自動部位拆解 (One Inference, Multiple Outputs)**：
  趁著引入 Human Parsing 跑一次就能掃描全身標籤的特性，讓後端自動拆解出 `upper` (上衣)、`lower` (下著) 與 `dress` (連身裙) 的獨立圖片並打包回傳。前端**完全免傳參數**，達到極致防呆與零運算浪費。

### 簡短執行步驟

1. 修改 v2 的 Gemini Prompt，要求多回傳一個 `presentation_mode` 欄位。
2. 實作 `HumanParsingRemoveBg` Strategy 策略（本地載入輕量級 SCHP ONNX 模型並進行推論）。
3. 於後端實作 Router 邏輯，依照 Gemini 預檢結果動態切換去背路徑。
4. 加入 OpenCV 後處理（GaussianBlur + Threshold），消除 Human Parsing 常見的邊緣鋸齒問題。

## v4 — 處理罕見但真實的 edge case 🟢（長期，1–2 個月）

### v4 要解決的問題

1. 單一模型在半透明、強反光、折疊堆疊等 edge case 偶爾失效。

### v4 預計怎麼做

| 問題 | 解法 |

|---|---|

| 1 | 多模型 ensemble（u2net + isnet + birefnet），或 SAM 2 互動式分割。 |

## v5 — 維運基礎建設（與 v2~v4 並行進行）⚪

不是新功能，而是長期穩定運作所需：

- 每階段加入耗時 log，掌握瓶頸。

- 針對大圖與併發保護記憶體（queue + 上傳上限）。

- 建立成功率/失敗率統計（SQLite 或等價儲存）。

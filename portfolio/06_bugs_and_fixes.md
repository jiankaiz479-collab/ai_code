## 故事 -1：閾值太寬鬆，fail loud 失靈（Robust v1 首次測試打臉）

- **時間**：2026-05-11 凌晨
- **觸發**：Robust v1 剛上線，拿暗光複雜場景測（地上躺照片）
- **預期**：rembg 切不出來 → 蒙版面積異常 → 系統回 1423 + diagnosis
- **實際**：系統回 200「成功」，結果圖**幾乎沒去任何背景**，只有邊緣有粉紅 / 橘色漸層暈染
- **檢查結果圖**：`processed_5abbb07d.png` vs 舊版 `modules_0055e839.png`（見 [08_demos/comparison_log.md](08_demos/comparison_log.md) 對照組 #1）

### 根因分析

我在 `RobustRemoveBg._mask_quality_check()` 設的閾值：
```python
DEFAULT_MASK_COVERAGE_MAX = 0.95   # 蒙版超過 95% 才判失敗
```

實際發生的事：
- isnet-general-use 對複雜暗光圖**過於保守**，把幾乎整張當前景
- 蒙版面積大概 90~94%
- **剛好沒撞到 95% 閾值**，系統判定「成功」
- alpha matting 的軟邊緣 + 周圍牆面顏色 → 詭異漸層暈染

### 學到什麼

1. **「fail loud」機制的閾值要嚴**：保守設計（寬鬆閾值）會讓失敗悄悄溜過
2. **設計 sanity check 時要假設 AI 的失敗模式是「全 true」或「全 false」**，不只是「很糟」
3. **AI 模型升級不一定更好**：isnet 對 u2net 在乾淨場景優勝、複雜場景反而更糟
4. **fail loud 是一個迭代過程**：上線後才知道閾值該設哪裡

### 修正方向

立即可做：
- `REMOVE_BG_MASK_COVERAGE_MAX=0.85` （閾值收緊）
- `REMOVE_BG_ALPHA_MATTING=false` （邊緣硬一點，更好判斷）
- 加新檢查：偵測「**蒙版邊緣的軟性過渡比例**」，過高也算失敗

長期：
- 對 `/fitting/modules` 加姿勢預檢查（這張圖的根本問題是姿勢不對，不該進到去背）

### 推甄面試金句

> 「我做 fail loud 機制時設了一個 95% 的閾值——蒙版超過 95% 才算失敗。實測才發現，**AI 模型實際輸出可能是 90%，剛好讓壞 case 溜過**。這讓我學到：**寫 sanity check 不是設一個合理閾值就好，要假設 AI 可能在任何百分比失敗——閾值應該以『要不要相信這個結果』為標準，不是『多糟才算糟』。**」

## 故事 0：假設打臉：iPhone 不一定是 HEIC，真正的問題是解析度與 EXIF

- **時間**：2026-05-10
- **情境**：規劃「現場拍照升級」時，我假設首要問題是 **iPhone HEIC 格式相容性**，主張要先做 HEIC 解碼
- **打臉**：使用者直接傳一張 iPhone 14 Plus 拍的截圖給我看 metadata，**格式是 JPEG**（不是 HEIC）
- **重新檢視真實 metadata**：
  - 解析度 **3024 × 4032 = 12MP**（rembg 跑很慢、吃滿記憶體）
  - 檔案 **3.5MB**（上傳頻寬負擔）
  - 長寬比 **3:4 直立**（與我們 1:1 標準化不合）
  - **EXIF rotation** 必定存在（iPhone 拍照都有）
  - 色彩 profile 可能是 **Display P3**（廣色域，與 sRGB 不同）
  - 場景：火鍋店、暗光、ISO 400、多人、半身入鏡

- **真正的問題優先序（修正後）**：
  1. **EXIF 旋轉** — 不論什麼手機都會踩
  2. **解析度上限** — 12MP 太大
  3. **色彩 profile 統一** — P3 → sRGB
  4. **格式正規化** — 把 HEIC/HEIF/JPG/PNG/WebP 全部統一處理（不是只擋 HEIC）
  5. **品質檢查** — 暗光 / 模糊 / 多主體

- **學到什麼**：
  1. **「常見錯」≠「最重要」**：HEIC 是常被提到的問題，但 iPhone 預設可選 JPEG，所以**未必是首要問題**
  2. **假設要用真實樣本驗證**：與其腦補「iPhone 多半是 HEIC」，不如直接拿一張使用者的真實照片看 metadata
  3. **問題優先序要由「100% 會發生」的事情往下排**：
     - EXIF rotation → 100% 會發生
     - 解析度過大 → 95% 會發生
     - HEIC 格式 → 60% 會發生（用戶可設 JPEG）
     - 廣色域 → 50%
  4. **推甄角度**：教授會問「為什麼做這個」，**回答「因為使用者直接秀照片打我臉」比「因為我先做了完整 survey」更真實**

- **影響改動**：
  - 把「HEIC 解碼」從「優先級 #1」降為「優先級 #4 - 格式正規化的子任務」
  - 把「EXIF rotation + resize」推上 #1 / #2
  - 後續實作會以「**所有格式 → 統一轉 RGB JPEG**」的設計，不單獨針對 HEIC

- **相關討論**：[02_architecture.md](02_architecture.md) 的「Strategy Pattern 重構」章節，原本對應的問題清單已修正

## 故事 6：使用者上傳「真人穿搭照」，rembg 卻把整個人切下來 (v3 雙軌路由的誕生)

- **時間**：2026-05-13
- **觸發**：系統上線後，使用者常常不是拍「平鋪在床上的衣服」，而是對著鏡子拍「自己穿著衣服」的樣子。
- **預期**：系統應該單獨切出照片中的衣服。
- **症狀**：`rembg` 把使用者的臉、手、腳連同衣服一起去背保留了下來，完全不符合「衣服單品」的規格。

### 根因分析
- `rembg` 的本質是「二元分割 (Binary Segmentation)」。在它的世界裡只有「背景(0)」和「前景(1)」。
- 對於人像模型來說，「人體」本來就是前景，它無法理解「衣服」與「人體」之間需要分離的語意界線。

### 我的假設 (按順序)
1. **假設 A（換 rembg 的衣服專用模型）** → 換成 `u2net_cloth_seg` 試試看 → **結果**：邊緣常常破碎，且還是無法穩定區分上衣和褲子。
2. **假設 B（讓 Gemini 給座標再切）** → 請 Gemini 輸出衣服的 Bounding Box 再用 OpenCV 裁切 → **結果**：LLM 給的 bbox 不夠精準，只能切出方形，背景依舊存在。
3. **假設 C（引入 Human Parsing 多類別分割 + LLM 路由）** → ✅

### 解法
1. 引入 **Human Parsing (如 SCHP)** 模型。它能進行像素級的 20 種類別分類，我只需提取 Label=5(上衣) 或 Label=9(褲子) 的像素，其他丟掉，**直接自帶去背效果**。
2. 為了不破壞對「平鋪圖」表現很好的舊版 rembg，我在前置的 Gemini 預檢層中加了一個 `presentation_mode` 欄位（讓 LLM 告訴我是平鋪還是真人穿著）。
3. 後端實作 **Router 模式**：平鋪走舊路徑，真人走新路徑。

### 學到什麼
1. **One Inference, Multiple Outputs**：AI 算力很貴，Human Parsing 跑一次推論，就能同時掃描出上衣和褲子，後端可以直接拆成兩張圖回傳給前端，體驗更好且不增加成本。
2. **架構的彈性**：先前實作的 Strategy Pattern 發揮了巨大作用，我只需要新增一個 Strategy，而不用改爛原有的程式碼。

- **相關 commit**：(待補)

### 推甄面試 1 分鐘 bug 故事講法

> 「我遇過一個痛點：使用者上傳對鏡自拍的穿搭照時，舊版去背模型會把整個人連同臉跟手一起切下來。因為對二元分割模型來說，人體就是前景。
> 
> 我的解法是引入『Human Parsing (人體部位分割)』模型，它能把像素分成 20 類，我只要把『上衣』那一類留著，其他包含人體都當作背景濾掉，直接一箭雙鵰完成部位提取跟去背。
> 
> 最有趣的是，我不能因此就把所有照片都交給新模型，因為如果使用者傳『放在床上的衣服』，新模型找不到人就會失敗。所以我設計了一個 **LLM Router**，讓 Gemini 預檢時順便告訴我這張照片是平鋪還是真人，後端再動態切換去背路徑。這個雙軌架構完美解決了多情境的問題！」

## 故事 7：真人穿搭去背後的「遮擋破洞」難題 (Garment Occlusion)

- **時間**：2026-05-14
- **觸發**：在實作完 v3 的 Human Parsing 雙軌路由後，成功把衣服從真人身上切下來了，但發現新問題。
- **預期**：切出來的衣服應該是一件完整的單品。
- **症狀**：如果照片中模特兒的手臂擋在衣服前面，切下來的衣服在肚子或胸口就會出現一個「手臂形狀的破洞」。

### 我的假設 (按順序)
1. **假設 A（丟給 Gemini VLM 幫忙畫）** → 把破洞衣服丟給 `gemini-1.5-pro`，叫它補全 → **結果**：VLM 是語言模型，無法做精準的像素級繪圖。
2. **假設 B（升級成生圖模型 gemini-3-pro-image-preview）** → 把原圖跟破洞衣服丟給生圖模型，請它還原 → **結果**：模型確實給出一件完整的衣服，但它是「整張重繪 (Redraw)」。新衣服的皺褶、光影跟邊緣像素完全改變了，無法用於後續要求像素對齊的 3D 重建。
3. **假設 C（使用專門的 Stable Diffusion Inpainting）** → 需要傳入破洞 Mask，鎖定原像素不變，只生成破洞處 → **結果**：技術上可行，但引入 SD 會讓系統架構變得極度龐大，且需要昂貴的 GPU 算力。

### 根因分析
Gemini 生圖模型的本質是「根據提示生成新圖」，而衣服遮擋還原需要的是「Masked Inpainting (遮罩修復)」，兩者的底層約束條件完全不同。要做到完美的像素級鎖定，必須依賴專門的 Inpainting 模型。

### 解法與產品取捨 (Graceful Degradation)
考量到系統複雜度與學生專案的算力限制，我沒有強行引入 SD，而是從「產品體驗」著手解題：
1. 將切出的不完整衣服，加上模糊平滑後，仍作為虛擬衣櫥的「預覽縮圖」。
2. 在前端 UI 加上提示：當 LLM Router 判定上傳的是「真人穿搭 (`worn_on_body`)」時，系統會溫柔提醒使用者：「真人照片提取的衣物可能因遮擋而不完整。若需進行高精度 3D 重建，建議上傳平鋪照 (Flat-lay)」。

### 學到什麼
1. **AI 能力的邊界**：生成新圖 (Generation) 跟局部修復 (Inpainting) 是兩個完全不同的技術領域，不能看 Gemini 會畫圖就逼它做修圖。
2. **工程師的產品思維**：當技術成本過高（需架設 SD）時，最好的解法不一定是硬幹技術，而是透過 UX 設計與路由機制來引導使用者，做 Graceful Degradation（優雅降級）。

### 推甄面試 1 分鐘 bug 故事講法

> 「在做真人去背時，我遇到一個難題：如果模特兒的手擋住衣服，切出來的衣服會有破洞。
> 
> 一開始我想：能不能把破洞的衣服丟給最新的 Gemini 生圖模型，叫它幫我補齊？但我實測後發現不行。因為 Gemini 是『整張重繪』，它會畫出一件皺褶跟光影完全不同的新衣服，這會導致後續的 3D 模組因為像素對不上而崩潰。這任務其實需要的是『Masked Inpainting (遮罩修復)』。
> 
> 業界要解這個通常會引入 Stable Diffusion Inpainting。但我評估了系統架構的複雜度跟算力成本，決定不在這個階段硬幹。我的解法是結合 LLM Router，當系統偵測到是真人穿搭時，會自動在前端提示使用者『若要最佳 3D 效果建議上傳平鋪圖』。這是我在 AI 能力邊界與系統成本之間做出的權衡。」

## 🐛 Case 0: 真實世界的髒資料 (HEIC 格式與現場拍照的現實差距)

**📍 發生情境**
專案初期測試去背功能時，我都使用網路上抓的乾淨衣服圖，系統運作完美。但當實際用 iPhone 14 Plus 拍攝測試時，去背結果卻徹底崩壞（邊緣抖動、主體錯位、甚至顏色不對）。

**💡 根本原因**
原本以為最大問題是「HEIC 格式」不支援，但經過排查，發現真正的優先序問題是：
1. **EXIF Rotation**：手機拍照其實是橫向記錄，靠 EXIF 標籤要求旋轉。Python (Pillow) 預設不處理，導致 AI 模型看到的是橫躺的衣服。
2. **解析度過大**：12MP 的巨大影像讓 rembg 處理極慢，且造成邊緣抖動。
3. **色彩 Profile**：iPhone 使用 P3 廣色域，但後端預設以 sRGB 處理，導致顏色產生偏移（例如紅衣變橘）。
4. **格式正規化**。

**🛠️ 解決方案**
在進入 AI 模型之前，加入一層強健的預處理 (`utils/image_io.py`)：
1. 使用 `ImageOps.exif_transpose` 強制套用旋轉。
2. 限制長邊不超過 2048px，進行等比縮放。
3. 偵測 ICC Profile 並強制轉換為 sRGB。

**🎓 學到的教訓**
「在實驗室裡 work 的模型，到了真實世界不一定能 work。」工程師的價值在於弭平 AI 模型與真實世界髒資料之間的落差。

## 🐛 Case 1: 為什麼更新了 `requirements.txt`，Docker 還是說找不到套件？

**📍 發生情境**
在實作「MinIO 歷史紀錄」功能時，我在 `requirements.txt` 新增了 `minio==7.2.5` 套件，接著執行了我寫好的自動化啟動腳本 `bash ai.sh`。結果系統啟動後，網頁直接崩潰報錯：
`ModuleNotFoundError: No module named 'minio'`

**🔍 排查過程**
1. 一開始以為是虛擬環境跑掉，但我確認我是用 Docker Container 在跑，套件理應被隔離在容器內。
2. 檢查 `Dockerfile`，裡面確實有寫 `RUN pip install -r requirements.txt`。
3. 仔細查看 `bash ai.sh` 執行時的終端機 log，發現了一行關鍵輸出：
   `✅ [1/2] 狀態：Docker 映像已存在，跳過建置。`

**💡 根本原因 (Root Cause)**
為了讓平常開發啟動更快，我在 `ai.sh` 寫了一段優化邏輯：
```bash
if [[ "$(docker images -q ${APP_IMAGE} 2> /dev/null)" == "" ]]; then
    docker build -t "${APP_IMAGE}" .
else
    echo "跳過建置"
fi
```
這導致了**只有在「電腦裡完全沒有這個映像檔」時才會觸發打包**。因為我之前已經啟動過系統，舊的映像檔還在，腳本就偷懶跳過了 `docker build`，所以新的 `requirements.txt` 根本沒有被讀進去，容器當然找不到 `minio`。

**🛠️ 解決方案**
- **短期解法**：手動執行 `docker build -t ai-code-app .` 強制 Docker 重新讀取配置並安裝新套件，再執行 `bash ai.sh` 啟動。
- **長期優化 (下一步)**：為 `ai.sh` 加入參數支援（例如 `bash ai.sh --build`），讓開發者可以顯式地要求腳本重新打包映像檔；或者改用 Docker Compose 的 `--build` flag 來讓機制更聰明。

**🎓 學到的教訓 (Takeaway)**
這讓我深刻理解了 Docker 開發中的兩個不同維度：
1. **掛載 (Volume)**：我們透過 `-v` 將本地 `ai_app/` 掛載進容器，所以修改 `.py` 程式碼時，Django 可以即時熱重載 (Hot-reload)，不需要重新打包。
2. **映像檔 (Image)**：當改變的是環境依賴（如 `requirements.txt` 或 `Dockerfile`）時，就**必須**重新 Build，因為這些東西是在映像檔建立的階段被寫死的。

> 「在開發自動化腳本時，我曾經遇到一個有趣的 Docker 陷阱。我寫了一個 `ai.sh` 腳本來一鍵啟動所有服務，為了加速，我設定『如果 Image 已存在就跳過 Build』。結果有次我新增了 MinIO 的套件，執行腳本卻一直報錯說找不到套件。

## 🐛 Case 2: 程式碼寫了 Model，但 Django 卻報錯找不到資料表？

**📍 發生情境**
當我寫好 `HistoryRecord` 模型並把邏輯接上 AI 模組後，打 API 測試卻得到 HTTP 500，錯誤訊息顯示：
`django.db.utils.OperationalError: no such table: ai_app_historyrecord`

**💡 根本原因**
Django 的 ORM 機制不會因為你寫了 `models.py` 就自動去改資料庫。因為系統是跑在 Docker 容器內，我只有重啟了容器，卻忘記對容器內的資料庫執行建立指令。

**🛠️ 解決方案**
透過 `docker exec` 對正在運行的容器下達遷移指令：
```bash
docker exec -it ai-container python manage.py makemigrations
docker exec -it ai-container python manage.py migrate
```

## 🐛 Case 3: Docker 內部網路「鬼打牆」(Connection Refused 與掛載盲點)

**📍 發生情境**
AI 處理完圖片要上傳 MinIO 時，日誌狂刷：
`MinIO 上傳失敗: Max retries exceeded (Connection refused to localhost:9002)`

**🔍 排查與解決過程**
1. **第一層坑（Docker 網路隔離）**：我原本設定 MinIO 網址為 `localhost:9002`，但在 Django 容器內的 `localhost` 是指它自己，根本找不到跑在另一個容器的 MinIO。
   * **解法**：修改 `settings.py`，將內部上傳網址改為 `ai-minio:9002` (Docker 內部 DNS)，前端讀取則維持 `localhost:9002`。
2. **第二層坑（掛載 Volume 遺漏）**：改完設定後重啟，**竟然還是報一樣的錯**！仔細檢查 `ai.sh` 才發現，我只把 `ai_app` 掛載進了容器，卻忘記掛載 `cv_testing_site`（設定檔所在目錄），導致容器一直讀到舊的設定檔！
   * **解法**：在 `docker run` 補上 `-v "$(pwd)/cv_testing_site:/app/cv_testing_site"`。
3. **終極防呆**：為了避免未來 `.env` 設定被誤蓋，我在 `StorageService` 內加了防呆機制：`if 'localhost' in endpoint: endpoint = 'ai-minio:9002'`，徹底根絕連錯網址的可能。

**🎓 學到的教訓**
微服務架構下，**「內部通訊 (Internal Network)」與「外部存取 (External Port Binding)」必須嚴格區分**；另外，Docker 掛載目錄時必須確保所有會變動的設定檔都有被 Volume 確實覆蓋。

## 🐛 Case 4: 系統 Port 被寫死，無法透過 `.env` 動態切換

**📍 發生情境**
原先為了開發方便，我在 `Dockerfile` 與啟動腳本中都寫死了 `8002` 埠。當專案需要部署測試時，雖然改了 `.env` 的 `RUN_PORT=8888`，但 Django 依然在 `8002` 啟動，導致網路對應失敗。

**💡 根本原因**
有三個地方綁死了 Port：
1. `manage.py runserver` 預設不給 Port 會跑 8000，原本我用 `sys.argv.append('8002')` 寫死。
2. `Dockerfile` 的 `EXPOSE` 和 `CMD` 寫死了 8002。
3. `ai.sh` 的 mapping 寫死了 `-p "${RUN_PORT}:8002"`。

**🛠️ 解決方案**
全面改為動態讀取環境變數：
1. 修改 `manage.py` 引入 `dotenv`，若無指定 Port 則自動讀取 `os.getenv('RUN_PORT', '8002')`。
2. `Dockerfile` 改用 shell 變數替換：`EXPOSE ${RUN_PORT:-8002}`。
3. `ai.sh` 修正 mapping 為 `-p "${RUN_PORT}:${RUN_PORT}"`。
從此只要修改 `.env` 就能一鍵改變全系統的 Port 綁定，大幅提升了自動化部署的彈性。

## 🐛 Case 5: 團隊協作時的 Port 衝突預防 (Port Collision Avoidance)

**📍 發生情境**
在部署 MinIO 服務時，我原本將容器內部的通訊埠設定為 `9000`（API）與 `9001`（Console）。雖然 Docker 網路有隔離機制，但團隊中的其他成員已經在他們的本機環境使用了這兩個 Port。為了徹底避免任何未來潛在的衝突或混淆，我決定主動進行調整。

**💡 根本原因**
微服務架構在團隊開發時，Port 的佔用是一個常見的隱患。雖然 Docker 只要對外 (-p) 不衝突即可運行，但在設定檔和腳本中留下與其他服務相同的 Port 號，極易造成協作時的誤解。

**🛠️ 解決方案**
將我們的 MinIO 服務全面避開 `9000` 與 `9001`：
1. 修改 `ai.sh` 中的啟動參數：`-p 9002:9002` 與 `-p 10092:10092`，並加上 `--address ":9002" --console-address ":10092"` 強迫 MinIO 更改內部 Port。
2. 更新 `settings.py`，將內部端點改為 `ai-minio:9002`。
3. 於 `StorageService` 更新防呆機制，將 `localhost` 強制導向 `ai-minio:9002`。

**🎓 學到的教訓**
在團隊協作中，**預防勝於治療**。除了依賴 Docker 的網路隔離外，主動在設定層面避開常用的 Port 號，能有效降低溝通成本與未來部署時發生靈異報錯的風險。

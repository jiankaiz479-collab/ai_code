## 故事 -1：閾值太寬鬆，fail loud 失靈（Robust v1 首次測試打臉）

- **時間**：2026-05-11 凌晨
- **觸發**：Robust v1 剛上線，拿暗光複雜場景測（地上躺照片）
- **預期**：rembg 切不出來 → 蒙版面積異常 → 系統回 1423 + diagnosis
- **實際**：系統回 200「成功」，結果圖**幾乎沒去任何背景**，只有邊緣有粉紅 / 橘色漸層暈染
- **檢查結果圖**：`processed_5abbb07d.png` vs 舊版 `modules_0055e839.png`（見 [08_demos/comparison_log.md](08_demos/comparison_log.md) 對照組 #1）

### 根因分析

我在 `RobustRemoveBg._mask_quality_check()` 設的閾值：
```python
DEFAULT_MASK_COVERAGE_MAX = 0.95   # 蒙版超過 95% 才判失敗
```

實際發生的事：
- isnet-general-use 對複雜暗光圖**過於保守**，把幾乎整張當前景
- 蒙版面積大概 90~94%
- **剛好沒撞到 95% 閾值**，系統判定「成功」
- alpha matting 的軟邊緣 + 周圍牆面顏色 → 詭異漸層暈染

### 學到什麼

1. **「fail loud」機制的閾值要嚴**：保守設計（寬鬆閾值）會讓失敗悄悄溜過
2. **設計 sanity check 時要假設 AI 的失敗模式是「全 true」或「全 false」**，不只是「很糟」
3. **AI 模型升級不一定更好**：isnet 對 u2net 在乾淨場景優勝、複雜場景反而更糟
4. **fail loud 是一個迭代過程**：上線後才知道閾值該設哪裡

### 修正方向

立即可做：
- `REMOVE_BG_MASK_COVERAGE_MAX=0.85` （閾值收緊）
- `REMOVE_BG_ALPHA_MATTING=false` （邊緣硬一點，更好判斷）
- 加新檢查：偵測「**蒙版邊緣的軟性過渡比例**」，過高也算失敗

長期：
- 對 `/fitting/modules` 加姿勢預檢查（這張圖的根本問題是姿勢不對，不該進到去背）

### 推甄面試金句

> 「我做 fail loud 機制時設了一個 95% 的閾值——蒙版超過 95% 才算失敗。實測才發現，**AI 模型實際輸出可能是 90%，剛好讓壞 case 溜過**。這讓我學到：**寫 sanity check 不是設一個合理閾值就好，要假設 AI 可能在任何百分比失敗——閾值應該以『要不要相信這個結果』為標準，不是『多糟才算糟』。**」

## 故事 0：假設打臉：iPhone 不一定是 HEIC，真正的問題是解析度與 EXIF

- **時間**：2026-05-10
- **情境**：規劃「現場拍照升級」時，我假設首要問題是 **iPhone HEIC 格式相容性**，主張要先做 HEIC 解碼
- **打臉**：使用者直接傳一張 iPhone 14 Plus 拍的截圖給我看 metadata，**格式是 JPEG**（不是 HEIC）
- **重新檢視真實 metadata**：
  - 解析度 **3024 × 4032 = 12MP**（rembg 跑很慢、吃滿記憶體）
  - 檔案 **3.5MB**（上傳頻寬負擔）
  - 長寬比 **3:4 直立**（與我們 1:1 標準化不合）
  - **EXIF rotation** 必定存在（iPhone 拍照都有）
  - 色彩 profile 可能是 **Display P3**（廣色域，與 sRGB 不同）
  - 場景：火鍋店、暗光、ISO 400、多人、半身入鏡

- **真正的問題優先序（修正後）**：
  1. **EXIF 旋轉** — 不論什麼手機都會踩
  2. **解析度上限** — 12MP 太大
  3. **色彩 profile 統一** — P3 → sRGB
  4. **格式正規化** — 把 HEIC/HEIF/JPG/PNG/WebP 全部統一處理（不是只擋 HEIC）
  5. **品質檢查** — 暗光 / 模糊 / 多主體

- **學到什麼**：
  1. **「常見錯」≠「最重要」**：HEIC 是常被提到的問題，但 iPhone 預設可選 JPEG，所以**未必是首要問題**
  2. **假設要用真實樣本驗證**：與其腦補「iPhone 多半是 HEIC」，不如直接拿一張使用者的真實照片看 metadata
  3. **問題優先序要由「100% 會發生」的事情往下排**：
     - EXIF rotation → 100% 會發生
     - 解析度過大 → 95% 會發生
     - HEIC 格式 → 60% 會發生（用戶可設 JPEG）
     - 廣色域 → 50%
  4. **推甄角度**：教授會問「為什麼做這個」，**回答「因為使用者直接秀照片打我臉」比「因為我先做了完整 survey」更真實**

- **影響改動**：
  - 把「HEIC 解碼」從「優先級 #1」降為「優先級 #4 - 格式正規化的子任務」
  - 把「EXIF rotation + resize」推上 #1 / #2
  - 後續實作會以「**所有格式 → 統一轉 RGB JPEG**」的設計，不單獨針對 HEIC

- **相關討論**：[02_architecture.md](02_architecture.md) 的「Strategy Pattern 重構」章節，原本對應的問題清單已修正

## 故事 1：Tripo 一直回 422，但照片明明很正常

> 範本 — 細節之後再補

- **症狀**：上傳清晰人像照，Tripo 一直回 `2008` (內容政策違規)
- **假設 1**：照片解析度太低 → 否（測 1024x1024 也失敗）
- **假設 2**：照片有違規內容 → 否（風景照也失敗）
- **假設 3**：API 版本參數錯誤 → ✅
- **根因**：model_version 設成已棄用版本
- **解法**：升 `TRIPO_MODEL_VERSION` 到 `v3.1-20260211`
- **學到**：上游錯誤訊息不一定準確，要對照官方文件版本表
- **影響改動**：建立 `_map_tripo_error()` 中央錯誤映射函數

## 故事 2：3D 紋理顏色偏移（紅衣變橘）

> 範本 — 細節之後再補

- **症狀**：原圖紅色 T-shirt，3D 模型出來是橘紅色
- **假設 1**：gamma 不對 → 否
- **假設 2**：texture_alignment 預設值不適合 → ✅
- **根因**：Tripo 預設 `texture_alignment=geometry`，會為了幾何對齊微調紋理
- **解法**：強制 `texture_alignment=original_image` + prompt 加 `no color shift`
- **學到**：第三方 AI 的預設值不一定符合你的場景需求，要看官方文件每個參數
- **相關 commit**：8ed2be7 `coloring second version`, d6c6af2 `fix prompt`

## 故事 3：開發測試把 Tripo 積分燒掉

> 範本 — 細節之後再補

- **症狀**：開發兩週 Tripo 積分用完，無法繼續測試
- **假設 1**：是測試流量太大 → ✅（每次改 view code 都要實際呼叫一次 Tripo 驗證）
- **根因**：沒有 mock 機制，每次按下「生成」都要燒積分
- **解法**：設計 `TRIPO_DEBUG_MOCK` 模式
- **學到**：**成本意識也是工程能力的一部分**，要把開發環境跟正式環境的 AI 呼叫分開
- **設計亮點**：mock 模式不只是「不呼叫」，還要**完全模擬真實流程的回應格式**，前端不需要區分

## 故事 5：Refine 階段加 1 倍時間

> 範本 — 之後補

- **症狀**：開啟 Refine 後生成時間從 35s 變 90s+
- **權衡**：品質 vs 時間
- **解法**：把 Refine 改成**選用**（`TRIPO_ENABLE_REFINE` 預設 false）+ 開發時關閉
- **學到**：feature flag 不只是 A/B test，也是「**成本 / 速度 / 品質**」的調整把手

## 推甄面試 1 分鐘 bug 故事講法（以故事 3 為例）

> 「我做這個專案中期遇到一個現實的問題：開發兩週就把 Tripo 積分燒完了。
> 因為每次改後端 view 的 code，都要實際呼叫一次 Tripo 驗證流程對不對——一次幾塊錢，兩週累積下來幾千塊就沒了。
>
> 我的解法不是『買更多積分』，而是設計一個 `TRIPO_DEBUG_MOCK` 環境變數。設為 true 時，**完全跳過 Tripo 呼叫**，直接讀本地預存的 GLB 檔——但**回應的 multipart 格式、JSON 結構、HTTP 狀態都跟真實流程一模一樣**，前端不需要做任何條件判斷。
>
> 這個經驗讓我覺得，**成本意識也是工程能力的一部分**。AI 服務跟以前的 SaaS 不一樣，每次呼叫都直接燒錢——所以開發環境跟正式環境的 AI 呼叫分離是必要的設計，不是過度工程。」

# 01. 問題定義

## 一句話定義

> 線上服飾消費者「不知道穿起來是什麼樣子」造成的退貨率與決策困難，能不能用 AI 自動生成「使用者本人穿這件衣服」的視覺化結果？

---

## 使用者痛點

| 角色 | 痛點 | 現有解法的不足 |
|---|---|---|
| 一般網購消費者 | 看完商品圖還是不知道自己穿起來像不像 | 試穿圖只用模特兒，跟自己身材無關 |
| 想開網店的小賣家 | 沒預算請模特拍試穿 | 找模特一次幾千塊 |
| 服裝設計師 | 改一件衣服要實拍才能看效果 | 來回打樣成本高 |

---

## 為什麼用 AI 而不是傳統圖形學？

| 方案 | 成本 | 真實感 | 上線速度 |
|---|---|---|---|
| 傳統建模 + 物理模擬 | ❌ 高（需要動畫師、TA） | ✅ 高 | ❌ 慢 |
| AI 圖像合成（Gemini / Diffusion） | ✅ 低 | ✅ 中~高 | ✅ 快 |
| AI 3D 重建（Tripo） | ✅ 中 | ✅ 中 | ✅ 快 |

**選擇：AI 為主、傳統圖形學僅做後處理**（例如方正標準化）。

---

## 為什麼需要 3D（而不只是 2D 試穿）？

2D 試穿 = 一張靜態合成圖，缺點：
- 看不到背面
- 衣服跟身體的物理互動只是「貼上去」
- 沒辦法給 VR/AR 用

3D 重建 = 一個 `.glb` 模型，優點：
- 可以**任意旋轉**（360° 看試穿效果）
- 為未來「布料物理模擬」鋪路（這是進研究所後想做的方向）
- 同一個模型可導入到遊戲引擎、Web AR

> **這也是我推甄想研究的方向之一：「神經網路與物理模擬如何結合產生真實布料行為」**

---

## 系統瓶頸釐清與優化探索（針對 3D 生成耗時）

> **面試防禦準備**：若委員提問「報告提到 3D 模型生成需時約 130 秒，為什麼這麼久？是你們寫的演算法太慢，還是硬體不夠？」

**📍 瓶頸責任歸屬（非本地端效能問題）：**
目前 3D 模型生成完整耗時較長（依情況需 90~130 秒不等），**並非我們系統後端架構或演算法效能低落**。主要瓶頸明確歸因於**第三方 API (Tripo 3D) 的物理與網路限制**：
1. **雲端服務當下的 Loading**：3D 生成是極度算力密集的任務，主要時間皆消耗在等待 API 伺服器排隊與推論（Long polling）。若遇伺服器尖峰時段，這部分延遲在我們端基本無解。
2. **請求任務的難度**：上傳圖片的特徵細節越多、服裝幾何越複雜，API 重建網格的運算時間也會相應增加。

**💡 優化探索：是否有探索過「效果不減但等待時間減少」的方案？**
為了解決這個瓶頸，我們在開發過程中做了以下實驗與權衡，確保在使用者體驗與品質間取得最佳平衡：
1. **捨棄冗餘精修，確保穩定性與時間雙贏**：
   我們透過自動化腳本實測數據發現，基礎的 `Draft` 階段生成約需 **140 秒**；若進一步開啟 `Refine`（精修）階段，不僅時間會無上限拉長，甚至經常因「網格過於複雜」被 Tripo 上游 API 拒絕，導致整筆任務失敗（拋出 `4422` 錯誤）。然而，`Draft` 模型在 Web 端的預覽上已經非常優異。因此我們**將系統設計為預設關閉 Refine 階段**（設定 `TRIPO_ENABLE_REFINE=false`），不僅大幅省下無謂的等待時間，更完美解決了 API 不穩定的痛點。
2. **調整 Payload 預處理與解析度下修實驗 (1024 vs 512)**：
   我們不直接把原始（可能是 4K 解析度且背景混亂）的圖片丟給 3D 引擎，而是在進入 3D 模組前，先由自家系統執行強制標準化（去背、置中、壓成 1024×1024 正方形），透過降低無效像素的運算難度來壓縮推論時間。
   **💡 為什麼不壓得更小（例如 512×512）來換取速度？**
   我們透過自動化腳本進行了 3 次對照實驗，實測數據完全推翻了「縮小圖片能加速」的假設，並從「時間」與「品質」兩個維度證實了 1024 的必要性：
   - **時間瓶頸在於伺服器排隊，耗時呈隨機震盪**：四次耗時實測分別為 `(1024: 168秒 vs 512: 132秒)`、`(1024: 124秒 vs 512: 180秒)`、`(1024: 178秒 vs 512: 164秒)`、`(1024: 126秒 vs 512: 125秒)`。這證明了生成時間在 120~180 秒間劇烈隨機震盪，完全取決於呼叫當下 Tripo 雲端 GPU 佇列的長短 (Long Polling)，而非圖片特徵維度。在客戶端壓縮解析度不僅無法縮短時間，反而會造成品質退化。
   - **網格面數與幾何資訊量流失**：1024 畫質生成的模型（Vertices: 53,274, Faces: 95,576）相較於 512 畫質（Vertices: 53,274, Faces: 94,064），多出了整整 1,512 個面。這代表 1024 畫質能讓 AI 演算法捕捉到更多微小的服飾幾何起伏，並用更密集的網格去描述。
   - **視覺與幾何特徵的致命差異**：
     - **裙擺與皺褶**：1024 畫質下模特兒手提的裙擺邊緣，摺皺的陰影轉折極為銳利立體，下擺波浪自然；反觀 512 畫質，同位置出現明顯的圓滑化與鋸齒退化，摺皺深度變弱（幾何特徵大機率退化變扁）。
     - **臉部與五官精細度**：1024 畫質的眼睛、鼻子立體感與臉部輪廓非常清晰；512 畫質的臉部線條則開始發糊，連頭髮邊緣的幾何邊界都出現不自然的平滑感。
   因此，我們確定將輸入解析度鎖定在 1024×1024，在確保品質的前提下，不再透過壓縮特徵來嘗試縮短耗時。
3. **從 UX 著手掩蓋等待感 (未來規劃)**：
   既然 3D 算力延遲是硬限制，我們規劃將任務轉為非同步架構（如 Celery）。前端先快速回傳耗時極短的 2D 試穿結果供使用者觀看，背景再慢慢生成 3D 模型，完成後推播通知，徹底從體驗上消解等待感。
